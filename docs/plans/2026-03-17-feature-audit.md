# Sunday Night Dynasty — Feature Audit: Design Doc vs. Implementation

**Date:** 2026-03-17
**Scope:** Every feature from `docs/plans/2026-03-16-dynasty-game-design.md` compared against actual code in `dynasty/dynasty/`.

---

## 1. Core Game Loop (Design Doc Section 2)

### IMPLEMENTED
- **SeasonPhase enum** — All 13 phases from the design doc are present: superBowl, proBowl, coachingChanges, combine, freeAgency, draft, otas, trainingCamp, preseason, rosterCuts, regularSeason, tradeDeadline, playoffs. (`Domain/Enums/SeasonPhase.swift`)
- **Phase ordering** — `WeekAdvancer.phase(after:)` correctly chains all offseason phases in calendar order: superBowl -> proBowl -> coachingChanges -> combine -> freeAgency -> draft -> otas -> trainingCamp -> preseason -> rosterCuts -> regularSeason. (`Engine/Simulation/WeekAdvancer.swift`)
- **18-week regular season** — `WeekAdvancer` advances weeks 1-18, transitions to playoffs at week 19. (`WeekAdvancer.swift:168`)
- **Playoff bracket** — Wild Card (wk 19), Divisional (wk 20), Conference Championship (wk 21), Super Bowl (wk 22) are all handled. (`WeekAdvancer.swift:184-218`)
- **Trade deadline marker** — Set at week 8. (`WeekAdvancer.swift:162-165`)
- **Bye weeks** — Assigned between weeks 5-14 with divisional separation. (`ScheduleGenerator.swift:263-299`)
- **New season bootstrap** — Resets records, generates schedule, advances season year. (`WeekAdvancer.startNewSeason`)
- **Career model** — Tracks playerName, role (GM/GM+HC), capMode, teamID, reputation, wins/losses, championships, yearsFired. (`Domain/Models/Career.swift`)
- **Advance Week button** — Dashboard has a single button that calls `WeekAdvancer.advanceWeek`. (`UI/Career/CareerDashboardView.swift:330-361`)
- **Game Summary sheet** — Shows after player's team game via `GameSummaryView`. (`CareerDashboardView.swift:56-67`)

### PARTIALLY IMPLEMENTED
- **Weekly cycle (Preparation -> Game Day -> Review)** — Only "Game Day" (simulation) and "Review" (GameSummaryView) exist. There is no preparation phase UI for setting depth chart or game plan before each game.
- **Game speed (1x, 2x, 4x, instant)** — MatchView supports 1x, 2x, 4x, and a "Skip to End" button, but this is only accessible from the MatchView itself, not integrated into the weekly advance flow. The player is shown a GameSummaryView sheet after advancing, not the live MatchView.
- **Trade deadline** — The phase is momentarily set and immediately reset to regularSeason (`WeekAdvancer.swift:163-164`). No trade UI or logic fires. The phase change is invisible.

### MISSING
- **Depth chart management** — No depth chart model, engine, or UI exists anywhere in the codebase.
- **Game plan setting** — No game plan model or UI. Play calling in `PlaySimulator.decidePlayCall` is purely situational with no user input.
- **Manual play-calling** — Design calls for selectable offense/defense play-calling and switching between simulation and manual mid-game. `MatchView` is view-only playback; the player cannot call plays.
- **Manage injured player statuses (weekly)** — No weekly injury management UI. Injuries exist on the Player model but there's no UI to set game-day statuses (active/inactive/IR).
- **Trade system** — No in-season trade UI or engine for player-for-player or player-for-picks trades outside the draft. `TradeOffer` model exists but is only used during draft pick swaps.
- **College scouting running in parallel during regular season** — `ScoutingEngine` generates prospects and reports but `WeekAdvancer` does not call it during the regular season. No weekly scouting report progression.
- **Preseason games** — The preseason phase exists but no games are generated or simulated during it.
- **OTAs / Training Camp activities** — These phases exist in the enum but `advanceOffseasonPhase` simply skips to the next phase with no logic.
- **Roster Cuts (to 53-man)** — Phase exists but no roster-cut logic or UI fires.
- **Pro Bowl / Awards** — Phase exists but no awards engine or UI.

### INTEGRATION GAPS
- **WeekAdvancer does not call any sub-engines during offseason phases** — When advancing through coachingChanges, combine, freeAgency, draft, otas, trainingCamp, preseason, or rosterCuts, the advancer simply sets `career.currentPhase = nextPhase`. It does not invoke `CoachingEngine`, `ScoutingEngine.simulateCombine`, `FreeAgencyEngine`, `DraftEngine`, `PlayerDevelopmentEngine.processOffseason`, `EventEngine`, `NewsGenerator`, or `OwnerSatisfactionEngine`. All of these engines exist but none are wired into the game loop.
- **WeekAdvancer does not call EventEngine during regular season** — Events are generated by `EventEngine.generateWeeklyEvents` but `WeekAdvancer.advanceRegularSeasonWeek` does not call it. Events exist in isolation.
- **WeekAdvancer does not call NewsGenerator** — News exists as an engine but is not invoked during week advancement.
- **WeekAdvancer does not call OwnerSatisfactionEngine** — Owner satisfaction never updates automatically.
- **WeekAdvancer does not call PlayerDevelopmentEngine** — No injury processing, no fatigue reset, no weekly injury checks happen via the game loop.
- **WeekAdvancer does not process contract expirations** — `contractYearsRemaining` is never decremented at season end.

## 2. Match Engine (Design Doc Section 3)

### IMPLEMENTED
- **Play-by-play simulation** — Full implementation across `PlaySimulator`, `DriveSimulator`, and `GameSimulator`. Each play goes through snap -> protection check -> QB decision -> pass/run/scramble -> result. (`Engine/Simulation/PlaySimulator.swift`)
- **Pass play phases** — O-line pass block vs DL pass rush + LB blitz -> sack check -> target selection -> accuracy/completion check -> interception check -> YAC. (`PlaySimulator.swift:182-392`)
- **Run play phases** — OL run block vs DL block shedding + LB tackling -> base yards + vision/elusiveness -> breakaway check -> fumble check -> TFL check -> safety/TD check. (`PlaySimulator.swift:396-535`)
- **Physical attributes affecting play** — Speed, strength, agility used in breakaway calculations, YAC, scrambling. (`PlaySimulator.swift` throughout)
- **Mental attributes affecting play** — Awareness and decision-making used in completion chance, morale modifiers. (`GameSimulator.swift:554-580`)
- **Position-specific attributes** — QB accuracy (short/mid/deep), pocket presence, scrambling; WR route running, catching; RB vision, elusiveness, breakTackle; OL run/pass block; DL pass rush, block shedding; DB man/zone coverage, ball skills; LB tackling, blitzing — all used in simulation. (`PlaySimulator.swift:860-960`)
- **Momentum** — Implemented with home field advantage (+0.1), decay rate, shifts on TDs (+0.15), turnovers (+0.20), big plays (+0.10), sacks (+0.05). Clamped to [-1, 1]. (`GameSimulator.swift:501-543`)
- **Fatigue** — Accumulates per drive for starters (+3), recovers for bench (+2), 30% reduction at halftime. (`GameSimulator.swift:582-606`)
- **Morale/Feel** — Mood-dependent players (feelPlayer, dramaQueen) suffer penalties when morale < 40. Consistent players unaffected. (`GameSimulator.swift:554-580`)
- **Clutch** — Q4/OT clutch bonus applied via `clutchQ4Multiplier`. (`GameSimulator.swift:572-578`)
- **Drive simulation** — Full down-and-distance management, clock management, quarter transitions, overtime rules (guaranteed possessions, sudden death). (`DriveSimulator.swift`)
- **Special teams** — Punt, field goal, extra point, two-point conversion, kneel, spike all simulated. (`PlaySimulator.swift:539-823`)
- **Box score** — Comprehensive: quarter scores, total/passing/rushing yards, first downs, 3rd down efficiency, turnovers, sacks, penalties, time of possession, drives. (`Domain/Models/League/BoxScore.swift`)
- **Player game stats** — Passing (yards/TDs/INTs/completions/attempts), rushing (yards/TDs/carries), receiving (yards/TDs/receptions/targets), defense (tackles/sacks/FF/INT), kicking (FGM/FGA), plus computed passer rating, YPC, YPR. (`Domain/Models/League/PlayerGameStats.swift`)
- **MVP selection** — Impact scoring with weighted TDs, yards, turnovers, defensive plays. (`GameSimulator.swift:871-911`)
- **Overtime** — Full NFL-style OT: coin toss, guaranteed possessions, first-TD-wins, sudden death, 10-minute cap. (`GameSimulator.swift:307-436`)
- **Game speed (1x, 2x, 4x)** — MatchView supports all three speeds plus skip-to-end. (`UI/Match/MatchView.swift:265-309`)

### PARTIALLY IMPLEMENTED
- **3D Match View (SceneKit)** — `SceneKitFieldView` and `FootballFieldScene` exist and render a field, but player models are not animated. The view shows a static field while the play-by-play text feed runs below it. No formations, routes, or player movement. (`UI/Match/SceneKitFieldView.swift`, `UI/Match/FootballFieldScene.swift`)
- **Injuries during games** — `PlayerDevelopmentEngine.checkForInjury` exists with durability/fatigue/age modifiers, but `GameSimulator` and `PlaySimulator` do not call it. In-game injuries never occur.
- **Scheme fit affecting performance** — `CoachingEngine.schemeFit` calculates fit scores, but `PlaySimulator` and `GameSimulator` do not use them. Scheme has zero effect on game simulation.
- **Drama (in-game events)** — Design mentions "player arguments, coach challenges, crowd effects (home/away)." Momentum provides home/away effect, but there are no in-game drama events.

### MISSING
- **Pre-snap reads** — Design specifies a pre-snap read phase. Implementation goes directly from play call to execution.
- **Offense and/or defense play-calling selectable independently** — No user play-calling at all. All play calls are AI-generated via `PlaySimulator.decidePlayCall`.
- **Switch between simulation and manual mid-game** — Not implemented. MatchView is purely a playback viewer.
- **Pause and make play-calls** — MatchView has play/pause for playback speed but no play-calling integration.
- **Color-coded player models with visible numbers** — SceneKit field exists but has no player models.
- **Formations and routes visible** — Not rendered.
- **In-game player arguments** — Not implemented.
- **Coach challenges** — Not implemented (NFL challenge flag system).

## 3. Player Model (Design Doc Section 4)

### IMPLEMENTED
- **Physical attributes (1-99)** — Speed, Acceleration, Strength, Agility, Stamina, Durability. (`Domain/Models/Player/PlayerAttributes.swift:5-27`)
- **Mental attributes (1-99)** — Awareness, Decision Making, Clutch, Work Ethic, Coachability, Leadership. (`PlayerAttributes.swift:31-53`)
- **Position-specific attributes** — All specified positions implemented:
  - QB: Arm Strength, Accuracy Short/Mid/Deep, Pocket Presence, Scrambling (`QBAttributes`)
  - WR: Route Running, Catching, Release, Spectacular Catch (`WRAttributes`)
  - RB: Vision, Elusiveness, Break Tackle, Receiving (`RBAttributes`)
  - TE: Blocking, Catching, Route Running, Speed (`TEAttributes`)
  - OL: Run Block, Pass Block, Pull, Anchor (`OLAttributes`)
  - DL: Pass Rush, Block Shedding, Power Moves, Finesse Moves (`DLAttributes`)
  - LB: Tackling, Zone Coverage, Man Coverage, Blitzing (`LBAttributes`)
  - DB: Man Coverage, Zone Coverage, Press, Ball Skills (`DBAttributes`)
  - K/P: Kick Power, Kick Accuracy (`KickingAttributes`)
- **Personality archetype** — Team Leader, Lone Wolf, Feel Player, Steady Performer, Drama Queen, Quiet Professional, Mentor, Fiery Competitor, Class Clown (last two are extras beyond design). (`Domain/Enums/PersonalityArchetype.swift`)
- **Motivation** — Money, Winning, Stats, Loyalty, Fame. (`Domain/Enums/Motivation.swift`)
- **Personality helpers** — `isDramaticInMedia`, `isMentor`, `isMoodDependent`, `isConsistent` computed properties. (`Domain/Models/Player/PlayerPersonality.swift`)
- **Hidden potential** — `truePotential` (1-99) on Player model. Not directly visible. (`Player.swift:21`)
- **Age curves** — Position-specific peak age ranges defined (QB 28-35, RB 24-28, WR 26-31, etc.). (`Domain/Enums/Position.swift:43-66`)
- **Player development** — Full offseason development engine with work ethic, coachability, playing time, coaching quality, age factor, potential ceiling. Physical vs mental weighting based on age. (`Engine/PlayerDevelopment/PlayerDevelopmentEngine.swift:18-90`)
- **Age regression** — Physical attributes regress first (speed, acceleration targeted), mental last (awareness, leadership protected). Durability specifically decreases with age and injury. (`PlayerDevelopmentEngine.swift:140-185`)
- **Mentor system** — Veterans with leadership > 75 and mentor/teamLeader archetype boost rookie mental attributes. Drama queens and low-leadership lone wolves are bad mentors. (`PlayerDevelopmentEngine.swift:195-231`)
- **Potential realization** — `updatePotentialRealization` adjusts truePotential based on scheme fit and morale. (`PlayerDevelopmentEngine.swift:242-272`)
- **Injury system** — `checkForInjury` with durability/fatigue/age/intensity modifiers. Severity tiers: minor (1-2 wks), moderate (3-6), major (7-16), season-ending (17-52). Permanent durability loss on healing (15% chance). (`PlayerDevelopmentEngine.swift:280-362`)
- **Retirement** — Age 35+ with increasing chance based on overall, injuries, durability. K/P play longer. (`PlayerDevelopmentEngine.swift:567-597`)
- **In-season experience** — Awareness and decision-making gain from games played/started, with rookie multiplier. (`PlayerDevelopmentEngine.swift:100-132`)
- **Overall rating** — Weighted average: physical 60%, mental 40%. (`Player.swift:42-46`)

### PARTIALLY IMPLEMENTED
- **Locker Room Impact** — Design specifies "Positive/negative/neutral effect on others." The mentor system handles positive/negative mentoring, and `EventOption.lockerRoomEffect` exists, but there is no persistent locker room chemistry score or team-wide morale propagation.
- **Scouted potential vs true potential** — `CollegeProspect` has both `truePotential` and `scoutedPotential`, and `ScoutingEngine.scoutProspect` reveals an estimate with error margin. However, once a prospect is drafted and converted to a Player, the true potential is immediately visible on the Player model (no gradual revelation over first seasons as design specifies).

### MISSING
- **Gradual attribute revelation for drafted players** — Design says "Player's 'true' attributes are revealed gradually over first seasons." Implementation converts prospect to player using true attributes immediately at draft time (`DraftEngine.convertToPlayer`).

### INTEGRATION GAPS
- **PlayerDevelopmentEngine.processOffseason is never called** — The full offseason pipeline (aging, development, mentoring, retirement) exists but `WeekAdvancer` never invokes it.
- **PlayerDevelopmentEngine.checkForInjury is never called during games** — The injury check system exists but neither `PlaySimulator` nor `GameSimulator` invoke it.
- **updatePotentialRealization is never called** — The method exists but nothing in the game loop calls it.
- **applyGameExperience is never called** — In-season experience gains exist but are never triggered.

## 4. Scouting and Draft (Design Doc Section 5)

### IMPLEMENTED
- **Scout model** — Position specialization, accuracy, personality read, potential read, experience, salary. (`Domain/Models/Scouting/Scout.swift`)
- **College prospect model** — Full model with true attributes (hidden), scouted attributes (revealed), combine results (40, bench, vertical, broad jump, shuttle, cone), interview/pro day completion flags, draft projection. (`Domain/Models/Scouting/CollegeProspect.swift`)
- **Scouting report** — prospectID, scoutID, overall/potential grades, strength/weakness/personality notes, confidence level. (`Domain/Models/Scouting/ScoutingReport.swift`)
- **Prospect generation** — 250-player draft class with position distribution, bell-curved attributes, random colleges (40 real schools). (`Engine/Scouting/ScoutingEngine.swift:56-103`)
- **Scout evaluation** — Accuracy determines error margin on overall/potential grades. Position specialization bonus (+10 accuracy). Personality read determines if archetype is correctly identified. Scout grade (A+ through F). Confidence level based on accuracy + experience. (`ScoutingEngine.swift:244-314`)
- **Combine simulation** — All 6 drills (40, bench, vertical, broad jump, shuttle, cone) correlated with physical attributes plus variance. "Combine warrior" / "bad tester" modifier (10% each). (`ScoutingEngine.swift:319-369`)
- **Pro Day simulation** — Similar to combine with slight "home field" boost; keeps best result. (`ScoutingEngine.swift:374-434`)
- **Interview** — Reveals personality and motivation based on scout's personalityRead. Work ethic and leadership hints. Can be misleading. (`ScoutingEngine.swift:440-484`)
- **Scout development** — Experience increments yearly; accuracy, personalityRead, potentialRead improve with diminishing returns. (`ScoutingEngine.swift:489-511`)
- **Draft order generation** — 7 rounds, 224 picks, reverse standings order with playoff team ordering and Super Bowl winner/loser at 31/32. (`Engine/Draft/DraftEngine.swift:20-92`)
- **AI draft logic** — Need-based with overall talent + positional need multiplier + potential bonus + QB premium. (`DraftEngine.swift:104-144`)
- **Prospect-to-player conversion** — True attributes used, rookie contract based on pick number. (`DraftEngine.swift:156-178`)
- **Pick value chart** — Classic NFL-style: pick 1 = 3000, declining steeply through round 1, moderately round 2, gradually rounds 3-7. (`DraftEngine.swift:189-442`)
- **Trade evaluation** — Compares total pick value on each side with player flat value. Returns ratio. (`DraftEngine.swift:215-237`)
- **AI trade-up offers** — 0-3 offers generated for high-value picks, package building from later picks, 85-130% value range, 25% chance. (`DraftEngine.swift:251-312`)
- **Draft UI** — Full draft view with on-the-clock card, player pick selection sheet, auto-pick, AI simulation with delays, completed pick feed, iPad big board sidebar, trade offer presentation. (`UI/Draft/DraftView.swift`, `DraftSelectionSheet.swift`, `DraftPickCard.swift`, `TradeOfferView.swift`)
- **Scouting Hub UI** — Hub view, scout team view, prospect list, prospect detail, big board. (`UI/Scouting/ScoutingHubView.swift`, `ScoutTeamView.swift`, `ProspectListView.swift`, `ProspectDetailView.swift`, `BigBoardView.swift`)
- **Draft risk** — Scouts can be wrong (accuracy attribute determines error margin), combine results can inflate/deflate, personality issues don't always show before draft. All implemented.

### PARTIALLY IMPLEMENTED
- **Scouting budget** — Scout has a `salary` field, but there is no team-level scouting budget model or budget enforcement.
- **Regional vs national scouts** — No distinction exists. Scouts have position specialization only.
- **Fall college season weekly reports** — `ScoutingEngine.scoutProspect` produces a single report. No progressive weekly improvement of reports as college season progresses.
- **Senior Bowl / shrine games** — Not implemented as distinct scouting phases.
- **Draft Week final big board** — The big board UI exists (`BigBoardView.swift`) but no final-board compilation step at draft week.

### MISSING
- **Scouting budget system** — No budget model, no spending limits on scouting operations.
- **Assigning scouts to watch college games** — No assignment UI or weekly scouting assignment system.
- **Reports arriving weekly and improving** — Scouting is a one-shot action, not progressive.
- **Future pick trades (next year's selections)** — `DraftPick.seasonYear` supports it structurally, but no UI or engine logic allows trading future-year picks.
- **Compensatory picks** — Not implemented.
- **Combine results can mislead (combine warrior vs game-smart)** — The combine warrior modifier exists in the engine (`combinePersonalityModifier`) but its effect is not surfaced to the player in a meaningful way (no "warning: combine results may be misleading" type messaging).

### INTEGRATION GAPS
- **ScoutingEngine.generateDraftClass is never called by WeekAdvancer** — Prospect generation exists but is not wired into the offseason flow. Prospects must be generated externally.
- **ScoutingEngine.simulateCombine is never called** — The combine phase exists but WeekAdvancer does not invoke it.
- **DraftEngine.generateDraftOrder is never called by WeekAdvancer** — Draft order must be generated manually.

## 5. Media, Owner, and Pressure (Design Doc Section 6)

### IMPLEMENTED
- **News system** — Full `NewsGenerator` with weekly and offseason news. Categories: gameResult, injury, trade, freeAgency, draft, coachingChange, playerPerformance, teamRanking, offFieldIncident, contract, retirement, award. Sentiments: positive, negative, neutral. (`Engine/Media/NewsGenerator.swift`)
- **Weekly headlines** — Power rankings, player of the week, injury reports, trade rumors, coaching hot seat, draft buzz, game results, contract news. 3-8 items per week. (`NewsGenerator.swift:57-120`)
- **Offseason news** — Phase-specific news for coachingChanges, combine, freeAgency, draft, otas, trainingCamp, rosterCuts. (`NewsGenerator.swift:125-266`)
- **Media market** — Small/Medium/Large with `mediaPressureMultiplier` (0.7/1.0/1.5) and `freeAgentAttraction` (0.8/1.0/1.3). Applied in owner satisfaction and free agency. (`Domain/Enums/MediaMarket.swift`)
- **Owner model** — Patience (1-10), spending willingness, meddling, prefersWinNow, satisfaction (0-100). (`Domain/Models/Team/Owner.swift`)
- **Owner satisfaction engine** — Updates based on win/loss, losing streaks, patience modifier, media market amplification, meddling factor, news sentiment, playoff/championship bonuses. (`Engine/Media/OwnerSatisfactionEngine.swift:9-104`)
- **Firing check** — Patience-adjusted critical/danger thresholds, firing probability 70-95% in critical zone, 30-50% in danger zone with multiple losing seasons. (`OwnerSatisfactionEngine.swift:110-137`)
- **Job offers after firing** — 1-5 offers based on reputation. Worse teams more likely to have openings. High-reputation coaches get offers from decent teams too. Descriptive offer text based on team record and market. (`OwnerSatisfactionEngine.swift:142-199`)
- **Owner satisfaction display** — Dashboard shows satisfaction percentage with green/yellow/red indicator. (`CareerDashboardView.swift:99-113`)
- **Owner Meeting UI** — Dedicated view accessible from dashboard. (`UI/News/OwnerMeetingView.swift`)
- **News UI** — Dedicated news view. (`UI/News/NewsView.swift`)

### PARTIALLY IMPLEMENTED
- **Coach hot seat reports** — Generated as news items (`NewsGenerator.generateCoachingHotSeat`) but only when teams have losses > wins and 4+ games played. No tracking of actual hot seat status.
- **Draft grades** — Design mentions "Grade: A- / F — Experts react to picks". No post-draft grade generation exists.
- **Bust/steal tracking** — Design mentions tracking draft picks over first seasons. Not implemented.
- **Fan patience** — Not a distinct metric. Media market implicitly handles this through `mediaPressureMultiplier`.

### MISSING
- **Pre-draft speculation and mock drafts** — No mock draft generation.
- **Media effects on player morale** — News sentiment affects owner satisfaction but does not directly affect player morale (especially Fame-motivated and Drama Queen types as design specifies).
- **Media effects on free agent attractiveness** — `MediaMarket.freeAgentAttraction` exists but `FreeAgencyEngine` does not use it.
- **Career reputation tracking after firing** — `Career.reputation` exists and `OwnerSatisfactionEngine.generateJobOffers` uses it, but reputation is never updated based on performance (it stays at the initial value of 50).

### INTEGRATION GAPS
- **NewsGenerator is never called by WeekAdvancer** — News generation exists but is not triggered during week advancement.
- **OwnerSatisfactionEngine.updateSatisfaction is never called** — Owner satisfaction never changes during gameplay.
- **OwnerSatisfactionEngine.checkFiring is never called** — The player can never be fired.

## 6. Off-Field Events and Drama (Design Doc Section 7)

### IMPLEMENTED
- **Event types** — All design categories covered: holdout, suspension, arrest, socialMediaIncident, retirementSpeculation, podcastControversy, manOfTheYear, voluntaryWorkouts, rookieImpresses, coachConflict, coordinatorInterview, veteranReturn, injurySetback, aheadOfSchedule, freakInjury, contractDispute, tradeRequest, teamChemistry. (`Engine/Event/EventEngine.swift:70-77`)
- **Event generation** — 0-2 events per week, probability driven by media market pressure and losing record. Weighted random pool based on player personalities, morale, contracts, roster composition. (`EventEngine.swift:87-139`)
- **Event templates** — Rich, multiple templates per event type with placeholder substitution ({playerName}, {teamName}, {coachName}). 2-3 options per event with morale, locker room, owner, and media effects. (`Engine/Event/EventTemplates.swift`)
- **Player decision consequences** — Each option has moraleEffect, lockerRoomEffect, ownerEffect, mediaEffect. Applied via `EventEngine.applyEventChoice`. (`EventEngine.swift:144-173`)
- **Event UI** — Full EventAlertView with event type badge, headline, related person info, description, response options with effect previews, confirm button. (`UI/News/EventAlertView.swift`)
- **Personality-driven probabilities** — Drama queens increase negative event weight, underpaid players increase holdout/contract dispute weight, unhappy players increase trade request weight. (`EventEngine.swift:186-222`)
- **Positive events** — Man of the Year, voluntary workouts, rookie impresses, veteran return, ahead of schedule, team chemistry — all with positive morale/locker room effects.
- **Coaching drama** — Coach conflict and coordinator interview events with dedicated templates.
- **Contract disputes** — Holdout and contractDispute types with multiple response options.

### PARTIALLY IMPLEMENTED
- **Franchise tag unhappiness** — Contract model has `franchiseTagged` field and `ContractEngine.franchiseTagValue` calculates tag value, but no event triggers from franchise tag status.
- **"Wants out" trade request** — Trade request events generate but there's no actual trade execution flow. The player sees the event and picks a response, but no trade can actually happen.

### MISSING
- **Offseason surgery/rehab progress tracking** — No persistent rehab tracking between weeks.
- **"Ahead of schedule" / "setback" events tied to actual injury recovery** — Events exist as random occurrences but are not mechanically linked to `injuryWeeksRemaining` changes.

### INTEGRATION GAPS
- **EventEngine.generateWeeklyEvents is never called by WeekAdvancer** — Events exist as a fully built engine but are never triggered in the game loop.
- **EventEngine.applyEventChoice effects are local only** — `lockerRoomEffect` is applied as a proxy to the single player's morale; it does not propagate to teammates. `mediaEffect` modifies coach reputation by +/-1 but does not generate news items or affect owner satisfaction.

## 7. Coaching Staff (Design Doc Section 8)

### IMPLEMENTED
- **Full hierarchy** — HC, OC, DC, STC, QB Coach, RB Coach, WR Coach, OL Coach, DL Coach, LB Coach, DB Coach, Strength Coach. All 12 roles. (`Domain/Enums/CoachRole.swift`)
- **Coach model** — firstName, lastName, age, role, offensiveScheme, defensiveScheme, playCalling, playerDevelopment, reputation, adaptability, personality, teamID, yearsExperience. (`Domain/Models/Coach/Coach.swift`)
- **Scheme enum** — Offensive: West Coast, Air Raid, Spread, Power Run, Shanahan, Pro Passing, RPO, Option. Defensive: 3-4, 4-3, Cover 3, Press Man, Tampa 2, Multiple, Hybrid. (`Domain/Enums/Scheme.swift`)
- **Scheme fit** — Comprehensive `CoachingEngine.schemeFit` evaluating every offensive scheme x position-attribute combination and every defensive scheme x position-attribute combination. Returns 0.0-1.0 with adaptability bonus. (`Engine/Simulation/CoachingEngine.swift:20-262`)
- **Coach development** — Age/experience increment, young coaches improve (1-3 attributes per year), veteran decline (20+ years), reputation changes based on team wins. (`CoachingEngine.swift:266-315`)
- **Hiring market** — `generateCoachCandidates` produces random candidates with age/experience distributions by role, correlated attribute ceilings, scheme assignments. (`CoachingEngine.swift:325-387`)
- **Coordinator poaching** — `checkCoordinatorPoaching` evaluates if coordinators get HC offers based on reputation, role, and team wins. (`CoachingEngine.swift:398-429`)
- **Player development bonus** — `coachDevelopmentBonus` considers coach playerDevelopment attribute, position-role match, scheme fit, personality compatibility, player coachability and work ethic. Returns 0.8-1.5 multiplier. (`CoachingEngine.swift:439-476`)
- **Personality compatibility** — Full 9x9 matrix of coach personality vs player personality producing bonuses/penalties. (`CoachingEngine.swift:537-622`)
- **Position-role matching** — `positionRoleMatch` correctly maps coach roles to player positions. (`CoachingEngine.swift:481-502`)
- **Coaching Staff UI** — Staff view grouped by HC/coordinators/position coaches, coach detail view, hire coach view with candidate comparison. (`UI/Staff/CoachingStaffView.swift`, `CoachDetailView.swift`, `HireCoachView.swift`)
- **Coach avatar** — Custom avatar component. (`UI/Common/CoachAvatarView.swift`)
- **League generator creates full coaching staffs** — 12 coaches per team at league generation. (`Data/Import/LeagueGenerator.swift:121-124`)

### PARTIALLY IMPLEMENTED
- **Coaching fit affecting development** — `coachDevelopmentBonus` is fully implemented and called by `PlayerDevelopmentEngine.developPlayer`, but since `processOffseason` is never called by the game loop, the bonus never actually applies.
- **GM Mode (not HC)** — `CareerRole.gm` exists and can be selected. However, there is no HC-GM relationship system, no HC disagreement on picks/roster decisions, and no drama from bad HC-GM relationships.

### MISSING
- **Coach careers (coaching tree)** — No tracking of where coaches go after leaving your staff.
- **HC may disagree on draft picks or roster decisions** — Not implemented.
- **Good HC-GM relationship bonus, bad = drama** — Not implemented.
- **Turnover affects team stability** — No stability metric.
- **Interview process / coach comparison UI for hiring** — `HireCoachView` exists with candidates but no structured interview process.

### INTEGRATION GAPS
- **CoachingEngine.developCoach is never called** — Coach development logic exists but is not invoked at season end.
- **CoachingEngine.checkCoordinatorPoaching is never called** — Poaching logic exists but is not invoked in the offseason flow.
- **Scheme fit is never used in game simulation** — `schemeFit` is calculated but `PlaySimulator`/`GameSimulator` don't reference it.

## 8. Contracts and Salary Cap (Design Doc Section 9)

### IMPLEMENTED
- **Two cap modes** — Simple and Realistic, chosen at career start via `CapMode` enum. (`Domain/Enums/CapMode.swift`)
- **Simple mode signing** — `ContractEngine.signPlayerSimple`: sets years, salary, team, debits cap. (`Engine/Contract/ContractEngine.swift:11-16`)
- **Simple mode cutting** — `ContractEngine.cutPlayerSimple`: frees cap, removes team assignment. (`ContractEngine.swift:19-24`)
- **Market value estimation** — Position multipliers (QB 1.8x, DE/CB 1.3x, etc.), age adjustment vs peak range, exponential curve rewarding elite talent. (`ContractEngine.swift:28-80`)
- **Realistic mode contract model** — Base salary (per-year array), signing bonus (prorated), guaranteed money, void years, no-trade clause, franchise tag. Cap hit and dead cap computed properties. (`Domain/Models/Contract/Contract.swift`)
- **Contract creation** — `ContractEngine.createContract` for realistic mode. (`ContractEngine.swift:85-104`)
- **Restructure** — Converts 80% of current base to signing bonus, spreading cost. (`ContractEngine.swift:109-137`)
- **Realistic mode cut** — Dead cap calculation with post-June 1 option (split across two years). (`ContractEngine.swift:145-163`)
- **Franchise tag** — `franchiseTagValue` calculates from top 5 positional salaries. (`ContractEngine.swift:169-174`)
- **Free agency engine** — Market generation from expired contracts, motivation-based salary modifiers (money +20%, winning -10%, loyalty -15%), desired years based on age, market interest based on overall. (`Engine/Contract/FreeAgencyEngine.swift`)
- **Free agent signing** — Works in both simple and realistic modes. Realistic auto-generates signing bonus and guaranteed money. (`FreeAgencyEngine.swift:86-126`)
- **AI free agency** — Sorts by overall, matches eligible teams by cap, randomly selects from top candidates, negotiates slight discount. (`FreeAgencyEngine.swift:131-169`)
- **Cap Overview UI** — Dedicated view. (`UI/Contracts/CapOverviewView.swift`)
- **Player Contract UI** — Dedicated view. (`UI/Contracts/PlayerContractView.swift`)
- **Contract Extension UI** — Sheet for extending contracts. (`UI/Contracts/ContractExtensionSheet.swift`)
- **Free Agency UI** — Full view with position filters, sort options, team cap display. (`UI/Contracts/FreeAgencyView.swift`)

### PARTIALLY IMPLEMENTED
- **AI teams follow the same rules** — `FreeAgencyEngine.simulateAIFreeAgency` uses simple mode for AI signings regardless of the player's cap mode choice, meaning AI teams don't have realistic contracts.
- **Player motivation affects negotiations** — Motivation multiplier affects asking price in FA market generation but not in direct contract negotiations with the player's team.
- **Salary cap growth** — Team has `salaryCap` field (default 255,000 = $255M) but there's no year-over-year cap growth mechanism.

### MISSING
- **Roster Bonus, Workout Bonus** — Not in Contract model.
- **Incentives (LTBE / NLTBE)** — Not modeled.
- **Option Years** — Not modeled.
- **No-Franchise Tag Clause** — Not modeled (no-trade clause exists).
- **Cap Rollover** — Not implemented.
- **Compensatory picks** — Not implemented.
- **Legal Tampering Period** — Not a distinct sub-phase within free agency.
- **Free agency waves** — No multi-wave FA simulation.
- **Agent personality** — Not modeled. Design mentions "tough negotiators vs fair dealers."
- **Bidding war mechanics** — `FreeAgent.marketInterest` exists but no actual competitive bidding UI or engine.

### INTEGRATION GAPS
- **FreeAgencyEngine is never called by WeekAdvancer** — Free agent market generation and AI signings exist but are not triggered during the freeAgency phase.
- **Contract years are never decremented** — `player.contractYearsRemaining` is set at signing but never reduced at season end, so contracts never expire.
- **Cap usage is not recalculated** — `team.currentCapUsage` is only modified at signing/cutting time. No annual recalculation.

## 9. Technical Architecture (Design Doc Section 10)

### IMPLEMENTED
- **SwiftUI** — All UI built with SwiftUI. Views in `UI/` directory. (`UI/MainMenu/`, `UI/Career/`, `UI/Roster/`, `UI/Schedule/`, `UI/Standings/`, `UI/Match/`, `UI/Staff/`, `UI/Contracts/`, `UI/Scouting/`, `UI/Draft/`, `UI/News/`)
- **SceneKit** — `SceneKitFieldView` and `FootballFieldScene` for 3D match view. (`UI/Match/SceneKitFieldView.swift`, `UI/Match/FootballFieldScene.swift`)
- **Swift Data** — All models use `@Model` macro: Career, League, Team, Player, Owner, Coach, Season, Game, Schedule, Contract, Scout, CollegeProspect, DraftPick. `DataContainer` creates the `ModelContainer`. (`Data/Persistence/DataContainer.swift`)
- **Pure Swift engine** — All engines are `enum` types with static methods, no framework dependencies: `GameSimulator`, `PlaySimulator`, `DriveSimulator`, `ScheduleGenerator`, `StandingsCalculator`, `WeekAdvancer`, `ContractEngine`, `FreeAgencyEngine`, `ScoutingEngine`, `DraftEngine`, `NewsGenerator`, `OwnerSatisfactionEngine`, `CoachingEngine`, `EventEngine`, `PlayerDevelopmentEngine`.
- **Architecture layers** — Clear separation: Domain (models + enums), Engine (pure logic), UI (SwiftUI views), Data (persistence + import).
- **Design doc engine list vs implementation:**
  - SimulationEngine -> `GameSimulator`, `PlaySimulator`, `DriveSimulator` (implemented)
  - DraftEngine -> `DraftEngine` (implemented)
  - ScoutingEngine -> `ScoutingEngine` (implemented)
  - ContractEngine -> `ContractEngine` + `FreeAgencyEngine` (implemented)
  - PlayerDevelopmentEngine -> `PlayerDevelopmentEngine` (implemented)
  - MediaEngine -> `NewsGenerator` + `OwnerSatisfactionEngine` (implemented)
  - EventEngine -> `EventEngine` + `EventTemplates` (implemented)
  - Additional: `CoachingEngine`, `ScheduleGenerator`, `StandingsCalculator`, `WeekAdvancer` (implemented beyond spec)

### PARTIALLY IMPLEMENTED
- **Roster Import/Export** — `LeagueGenerator` creates randomized leagues with 32 teams, 53-man rosters, owners, coaching staffs. `NFLTeamData` has all 32 real NFL teams with correct conferences, divisions, and market sizes. `RandomNameGenerator` produces player names. However, there is no JSON import/export system, no roster pack loading, and no community sharing.
- **ViewModels / Presentation layer** — The design specifies a presentation layer with ViewModels. The codebase uses SwiftUI views with `@State`, `@Query`, and `@Environment(\.modelContext)` directly — no dedicated ViewModel layer exists.

### MISSING
- **JSON-based roster import/export** — Not implemented.
- **Community roster pack sharing** — Not implemented.
- **Save/Load system** — Swift Data persists automatically, but there's no explicit save game management UI (multiple saves, save slots, etc.).

## Critical Path: The Integration Gap Problem

The single most impactful issue is that **WeekAdvancer is a bare-bones state machine**. It handles game simulation and phase transitions but calls almost no sub-engines. The following engines exist as fully functional code but are never invoked by the game loop:

1. `PlayerDevelopmentEngine.processOffseason` — aging, development, mentoring, retirement
2. `EventEngine.generateWeeklyEvents` — off-field drama and player decisions
3. `NewsGenerator.generateWeeklyNews` / `generateOffseasonNews` — dynamic news feed
4. `OwnerSatisfactionEngine.updateSatisfaction` — owner pressure system
5. `OwnerSatisfactionEngine.checkFiring` — career stakes
6. `FreeAgencyEngine.generateFreeAgentMarket` / `simulateAIFreeAgency` — offseason signings
7. `ScoutingEngine.generateDraftClass` / `simulateCombine` — prospect pipeline
8. `DraftEngine.generateDraftOrder` — draft setup
9. `CoachingEngine.developCoach` / `checkCoordinatorPoaching` — coaching lifecycle
10. Contract year decrements — contract expiration

Wiring these into `WeekAdvancer.advanceWeek` and `advanceOffseasonPhase` would activate roughly 40-50% of the features currently marked as "integration gaps" or "partially implemented," dramatically increasing the game's depth without writing new engine code.
